# 00 · 设计理念与架构总览

> 这是系列的起点。本篇不写一行能跑的代码，只回答一个问题：**为什么要从零搭一个 agent harness，而不是写个 while 循环就行了。**

## 1. 这层在做什么

把整个系列的「地图」摆出来。后面 15 篇每深入一个模块时，你都能回到这张地图，知道它在哪儿、为什么存在。

一句话定位：我们要搭的不是聊天机器人，是一个**能在生产环境稳定运行的 agent 框架**。这两者的差距，就是本系列的全部工作量。

## 2. 不这样会坏什么

直接看一下「最朴素的 agent」长什么样：

```python
while True:
    reply = llm.invoke(messages)
    if not reply.tool_calls:
        break
    for call in reply.tool_calls:
        messages.append(run_tool(call))
```

它确实能跑。但放到生产里，它会在四个地方坏掉——而每个坏掉的地方，都对应后面一篇要解决的问题。

### 坏点一：中断与恢复

用户发了一条消息，agent 跑了 30 秒；用户网断了，或者关掉了浏览器。你的 while 循环还在跑，但没人能看结果了。用户重开浏览器发新消息，代码该怎么接上？你可能说「把 messages 存起来」——存在哪？什么时候存？中途崩了怎么办？

这是 **checkpointer** 要解决的：在每个节点执行完自动持久化状态，崩了能恢复，断线能续接。不写你自己的存盘逻辑。（见第 01 篇）

### 坏点二：流式输出

while 循环得等整轮跑完，可能 30 秒后才给用户看到结果。LLM 第一个 token 出来的瞬间就该能推给前端，文字逐字落下。

这是 **流式协议** 要解决的。生产级的体验，不能让用户盯着空白转圈等半分钟。（见第 12 篇）

### 坏点三：并发数据合并

两个子任务都想往 messages 里追加结果——最后一个写入会覆盖另一个，静默丢数据。

这是 **reducer** 要解决的：给状态字段配「合并规则」，并发写入有明确语义而不是靠运气。（见第 01 篇）

### 坏点四：横切关注点变成泥球

你需要在每次 LLM 调用前后插入：安全审计、错误兜底、死循环检测、上下文压缩、token 计费、记忆注入……在 while 循环里只能硬写进主循环，于是主循环变成这样：

```
check guardrails → inject memory → inject images →
check token limit → loop detected? → clarification requested? →
update title → log → to memory → ...
```

这段代码致命：**改一个功能要在巨函数里找到位置，极容易误伤别的；想单独测「死循环检测」必须构造满足所有前置的完整场景；想关掉记忆功能没法简单关，因为它和别的逻辑交织在一起。**

这是 **中间件链** 要解决的——把每个横切关注点变成一个独立类，各管各的事，按顺序组合。（见第 03 篇）

> 这四个坏点合起来就是一句话：**while 循环能让你 demo 跑起来，但要让它「生产可用」，你最终会重写一个残缺的 LangGraph。**

## 3. 关键概念

### 三种能力的分层

一个 agent 系统天然分三层，混在一起就乱：

| 层 | 关心什么 | 本系列篇目 |
| --- | --- | --- |
| 内核 | 状态、模型、中间件、工具——agent 怎么「想」和「做」 | 01–09 |
| 平台 | 怎么被调用：REST API、嵌入式、可观测、测试 | 10–13 |
| 外围 | 用户怎么看到它：前端、部署、实战串接 | 14–16 |

### 四个贯穿全书的设计哲学

这两页只是先布好，后面每篇都会在某个具体模块上落地它们：

1. **配置驱动 + 反射加载**：把「用哪个模型/工具」从代码里搬到配置文件，靠类路径字符串在运行时反射实例化。换模型不改代码；支持新 provider 不改框架源码。（第 04、05 篇）
2. **中间件链**：横切关注点各管各的，独立实现、独立测试、按序组合。（第 03 篇）
3. **线程隔离 + 虚拟路径**：每个对话线程一块独立目录，agent 看到的是虚拟路径（如 `/mnt/user-data/workspace`），框架背后映射到真实目录。两重收益——安全（限制越界访问）、多租户（互不干扰）。（第 07 篇）
4. **可观测性**：一个跑着的 agent 必须能回答「这次用了多少 token、哪个工具最慢、今天有没有死循环」。不是事后加，是从设计就考虑。（第 13 篇）

### 一次请求的生命周期（先扫一眼）

```
用户发消息
  → 【准备】 分配沙箱、建线程目录、注入上传文件
  → 【调用LLM前】 注入记忆/图片、检查 token 预算
  → 【调LLM】 决策：直接回答 还是 调工具
  → 【工具执行】 权限校验 → 执行 → 审计 → 结果写回 → 回到「调LLM」
  → 【结束】 生成标题、提取记忆、记录 token 用量
  → SSE 实时推流到前端
```

后面每篇，都是这张图上的某一环。

## 6. 踩过的坑 + 进阶

**进阶（领域里更深的做法）**：成熟的 agent 框架会把「框架层」和「应用层」**物理隔离**——框架能被独立打包发布，不依赖任何应用代码；这条边界不是口头约定，而是写成一个测试，CI 每次提交都扫描，一旦框架代码 import 了应用代码就立刻失败。为什么用测试而不是代码审查？因为人会累、会赶 deadline 放行「就这一次」，测试不会。把架构约束固化成可执行检查，是大型系统长期不腐烂的关键。
